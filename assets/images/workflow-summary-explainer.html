Behind the Feature

How AI Turns Workflow Logic into One Sentence

A look at how we use structured prompting to make complex recruitment automations instantly understandable.

Scroll to explore

Setting the Stage

First, the world this lives in

SearchOS is a recruitment platform. Companies use it to manage hiring pipelines — tracking candidates through stages like Leads → Approved → Intros Pending → Interviewing → Hired.

At each stage, workflows can fire automatically. A workflow is a set of rules: "When a candidate enters this stage, do these things." Things like requesting a warm introduction, sending an outreach email, waiting a few days, then following up.

The problem

Workflows are built visually as tree-like flowcharts with triggers, conditions, waits, and actions. Powerful for the person who built them — but when a teammate glances at a stage column and sees "1 workflow active", they have no idea what it actually does without opening the editor and reading every node.

The Journey

From workflow tree to one sentence

Here's what happens, step by step, from the moment a workflow is saved to the moment a user reads its summary.

01

Someone saves a workflow

A recruiter builds or edits a workflow in the visual editor — adding triggers, conditions, email sends, waits, stage moves. When they save, a background job is automatically queued. No one clicks "generate summary." It just happens.

02

The system reads the workflow tree

The job walks the workflow's node tree recursively — trigger, actions, conditions, children. But here's the key: it translates internal IDs into human words before anything reaches the AI. A stage UUID becomes "Approved." A person ID becomes "Sarah Chen." Template IDs are stripped entirely. The AI never sees technical identifiers.

03

A structured prompt goes to the AI

Two parts are sent: a system prompt (fixed instructions that never change) and a user prompt (the workflow data, different every time). The system prompt gives the AI recruiting context, tells it to write one sentence starting with "When," and shows good vs. bad examples. The user prompt contains the tree structure for this specific workflow.

04

The summary is saved to the database

The AI returns one sentence. It's saved directly on the workflow record alongside a timestamp. This means the summary is computed once when the workflow changes — not every time someone looks at it. Fast reads, no waiting.

A user clicks the workflow icon

On the project board, each stage column shows a small workflow indicator. Click it, and a popover appears with the workflow name and its pre-generated summary. The AI call already happened in the background — this is just reading a text field from the database. Instant.

Inside the Prompt

What the AI actually sees

The prompt has two parts. Hover over the yellow highlights to see what changes for each workflow.

prompt sent to AI model
System Prompt — stays the same every time

You are describing workflows for a recruiting
company that automates candidate outreach.

Describe what the workflow does in ONE sentence.
Start with "When" and focus on the business outcome.

// includes: domain vocabulary, good & bad examples,
// interpretation rules (e.g. "multiple emails = nurture sequence")
User Prompt — different for every workflow

Workflow structure:

  trigger: candidate_moved
    - from_stage_name: Leads
    - to_stage_name: Approved

  action: wait
    - duration: 2 hours

  condition: has_connector
    - matching_mode: relationship_score

    action: request_intro
      - stage_name: Intros Pending

    action: send_linkedin_inmail
      - stage_name: Cold Outreach

Write ONE sentence describing what this workflow does.
Start with "When" and focus on the business outcome.

The dynamic parts are the key idea. The same system prompt is reused for every workflow. But the user prompt changes completely — different triggers, different actions, different stage names, different branching paths. The AI writes a unique summary each time because it's seeing unique data.

What Makes This Work

The design decisions that matter

⚙

Translate before prompting

Internal IDs are resolved to human-readable names before anything reaches the AI. The model never sees a UUID.

✍

Constrain the output

"ONE sentence. Start with When." Tight constraints prevent the AI from over-explaining. Good/bad examples teach by showing, not telling.

⚡

Pre-compute, don't live-generate

Summaries are generated when workflows change, not when users look at them. Reads are instant; the AI latency is invisible.

01

Give the AI domain context

The system prompt explains the recruiting domain — warm intros, cold outreach, nurture sequences. Without this, the AI would describe mechanics instead of business meaning.

02

Separate fixed from dynamic

The system prompt (instructions, examples, vocabulary) stays constant. Only the workflow structure changes per call. This makes the output predictable and consistent.

03

Include an interpretation guide

The prompt maps technical patterns to business concepts: "multiple emails in sequence = nurture campaign." This bridges the gap between what the data says and what users understand.

04

Fail gracefully

If the AI call fails, the summary is simply missing. The UI shows "No description available" in italic. The feature never blocks the page or shows an error to the user.

If You Know Claude Code

This is the same pattern, applied differently

If you use Claude Code with a system prompt (like a CLAUDE.md file or custom instructions) and then ask it questions, you're already doing this exact pattern:

In Claude Code
CLAUDE.md = System prompt
Persistent instructions that shape every response. Your project rules, coding conventions, preferences.
In This Feature
Recruiting context = System prompt
Persistent instructions about the domain. "You describe workflows for a recruiting company..."
In Claude Code
Your question = User prompt
Changes every time. "Refactor this function" or "Add tests for X." Different data, same rules.
In This Feature
Workflow tree = User prompt
Changes for every workflow. Different triggers, actions, conditions. Same formatting rules.

The difference is that in Claude Code, you write the user prompt by typing in the chat. In this feature, the code writes the user prompt automatically by reading the workflow tree from the database, translating IDs to names, and formatting it as structured text. The AI never sees a chat interface — it receives one API call with two parts and returns one sentence.
